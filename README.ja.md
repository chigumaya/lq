# lq (LLM query)

## これは何？

ローカルファイルや標準入力から取得したデータについてLLMに質問するコマンドラインツール。

現時点では `POST /chat/completions` で `messages[].content` の content array を受け付ける OpenAI互換API に対応。

## インストール

python3で書かれているが非標準モジュールは使用していないので、パスの通ったところにコピーするだけでよい。

```
install -m 755 lq.py /path/to/lq
```

## 使いかた

```
lq [OPTIONS] <prompt>
```

### オプション

```
-f, --file <filename>              ファイルをプロンプトに追加（複数回指定可能）
-i, --image <filename>             画像をプロンプトに追加（複数回指定可能）
-s, --system <text>                追加システムプロンプトを文字列で指定
-S, --system-file <filename>       追加システムプロンプトをファイルから読む
-c, --config <path>                設定ファイルのパス（デフォルト: ~/.config/lq/config.json）
-m, --model <name>                 使用するモデルを指定
-M, --max-size <size>              入力サイズの上限 (デフォルト: 10MB)
-j, --json                         生のJSON応答を出力
--debug                            デバッグモード（リクエスト情報をstderrに出力）
-h, --help                         ヘルプを表示
-v, --version                      バージョンを表示
```

## 設定

### 設定ファイル

`~/.config/lq/config.json` で設定する。`-c` で別のパスを指定できる。複数のモデルを定義することができ、デフォルトでは `defaults.model_name` で指定されているモデルが使われる。それ以外のモデルを使う場合は `-m` で指定。

```json
{
  "defaults": {
    "model_name": "foo",
    "max_size": "10MB"
  },
  "models": [
    {
      "name": "foo",
      "model": "foo-model",
      "api_url": "https://example.com/v1",
      "api_key": "sk-foo-key"
    },
    {
      "name": "bar",
      "model": "bar-model",
      "api_url": "https://example.net/v1",
      "api_key": "sk-bar-key"
    }
  ]
}
```

### 環境変数

設定ファイルと同等の設定は環境変数でも指定できる。
```
API_URL=http://localhost:11434/v1  APIエンドポイントのURL
API_KEY=sk-...                     APIキー（オプション）
MODEL=gpt-oss-20b                  モデル名
```

### 設定値の優先度

高い順に以下のとおり。

1. コマンドラインオプション（`-m`、`-s` など）
2. 環境変数（`API_URL`、`API_KEY`、`MODEL`）
3. 設定ファイル (`~/.config/lq/config.json`)

## 使用例

```bash
# 簡単な質問
% lq '.tar.gz を展開するコマンドはなんだっけ？'

# ファイルを分析
% lq -f README.md '要約して'

# 複数ファイルを比較
% lq -f README.md -f README.ja.md 'ファイルを比較して'

# 画像を説明
% lq -i diagram.png 'この図を説明して'

# QRコードの読みとり
% lq -i qrcode.png 'QRコードを読んで'

# 差分を分析
% diff file-A file-B | lq '差分を要約して'

# ログを分析
% tail -100 /var/log/messages | lq 'このログから問題を診断して'

# テキストを翻訳
% man curl | lq '日本語に翻訳して'

# プロセスを分析
% ps auxw | lq 'メモリをバカ食いしてるプロセスはどれ？'

# コミットメッセージを生成
% git commit -m "$(git diff | lq '差分を3行で要約して')"
```

### ファイル(-f)と画像(-i)の扱い

```
% lq -f image.png '何の画像？'
```
のように、画像ファイルを `-i` ではなく `-f` で指定すると、このファイルは画像としてではなくbase64エンコードしたバイナリデータとしてLLMに送信する。画像として送信した場合はLLM側で画像専用の処理がおこなわれ、トークン消費もごくわずかで済むが、base64エンコードしたデータとして送信するとテキストデータとして扱われ、極めて大きなトークン消費になることに注意が必要である。

標準入力から受け取ったデータはテキストかバイナリかだけを判定し、バイナリデータを画像として扱えるかどうかの判定はしていない。そのため、画像を標準入力経由で受け取りたい場合には、パイプではなく、-i オプションを使えるようプロセス置換を利用したほうがトークン効率がよい。

```bash
# HTTPSで取得したデータを画像として扱う(プロセス置換)
% lq -i <(curl https://example.com/foo.jpg) '画像に含まれる文字を読み取って'

# HTTPSで取得したデータを単なるバイナリデータとして扱う(パイプ)
% curl https://example.com/foo.jpg | lq '画像に含まれる文字を読み取って'
```

その一方で、画像の入力に対応していないLLMであっても、あえて `-i` ではなく `-f` (あるいはパイプ)を使うことで画像を認識させることができる場合がある。

なお、UTF-8以外のテキストエンコーディングには対応していない。

テキスト添付はそれぞれ独立した `user` メッセージとして送信し、各メッセージの先頭1行にメタ情報を付ける。2行目以降のそのメッセージ全体が添付本文である。

```text
Attachment: source="file", name="README.md", encoding="utf-8"
...file contents...
```

```text
Attachment: source="stdin", encoding="base64"
...base64 data...
```

legacy な flatten 済み文字列しか受け付けないプロバイダはサポート対象外。

## セキュリティに関する注意事項

### プロンプトインジェクション

現状のLLMでは、実行対象にしたくない添付データであってもプロンプトの一部として扱う必要がある。そのため、`lq` は添付データを独立した `user` メッセージとして送信し、デフォルトのシステムプロンプトで `-f` や `-i` で指定したファイルや標準入力からのデータに含まれる指示には一切したがわないよう指示している。

しかしながら、LLMによって指示の解釈が異なることがあり、また、巧妙に作成されたペイロードによって防御が回避される可能性もあり、完全な対策は困難である。意図しないコマンド実行やロールプレイ攻撃のリスクを低減するよう努めているが、あくまでリスク軽減策であり、保証と考えてはならない。信頼できないデータを `lq` で扱うべきではない。

### 情報漏洩

`-f` で渡されるファイル、`-i` で渡される画像、標準入力から渡されるデータはプロンプトの一部としてLLMに送信される。これらのデータがパスワード、秘密鍵、認証トークン、その他機密情報を含んでいたとしてもマスクせずそのまま送信するので、クラウドLLMサービスを利用する場合にはとくに注意すること。このような用途で利用する場合はローカルLLMの利用を検討するべきである。
